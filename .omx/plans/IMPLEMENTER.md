## [작업 단위 흐름]

작업 단위 = 한 번 산출 후 멈추는 단위. 컴포넌트 안에서 implementer가 구분.

### 작업 단위 들어갈 때

다음 셋을 view 한 뒤 작업 시작:
1. 지금 다루는 spec 절 (해당 컴포넌트 §)
2. 차용이 있으면 차용 출처 파일의 인용 라인
3. 이번 작업이 의존하는 이전 결정이 있으면 해당 decisions/NNN-*.md

### 작업 단위 종료 시

다음 순서로 처리:
1. 산출물 작성 — 다음 셋 중 하나:
   - 코드 (구현)
   - 결정 노트 (L2)
   - OPEN 산출 (L1) 또는 충돌 보고
2. 채팅에 한 줄 요약 + "다음 작업: <무엇을 할 차례>"
3. 멈춤. 자동 진행 금지.

### 작업이 막힐 때

다음 다섯 중 하나로 분류 후 그에 맞는 산출:

| 상황 | 처리 |
|---|---|
| spec이 모호 또는 결정 없음 | OPEN-L1 산출 → open/NNN-*.md, 사람 결정 대기 |
| spec과 코드가 충돌 | 채팅에 충돌 보고, 둘 중 어느 쪽이 맞는지 추측 금지, 멈춤 |
| 차용 코드가 spec 가정과 다름 | 충돌로 처리 (위와 같음) |
| 자기가 결정 가능 (영향이 컴포넌트 안에 갇힘) | L2 결정 노트 → decisions/NNN-*.md, 작업 진행 |
| 그 외 — 무엇이 막혔는지 모름 | 채팅에 상태 보고, 사람 판단 대기 |

### 흐름 적용 범위

- 모든 작업 단위에 적용. 컴포넌트 첫 작업이든 마지막 작업이든.
- 작업 단위 안의 가운데(어떻게 짜는가)는 implementer 판단. 들어갈 때/종료 시/막힐 때 셋만 이 절로 박힘.
- "계획/테스트/검증" 같은 단계 강제 없음. 작업 성격에 맞게 implementer가 결정.

[OPEN 분류]
L1 (critical) — 후속 결정 다수 의존, roll back 비용 큼
  예: 구현 substrate, event log storage 모델, core schema 1차, 차용 대상 변경
L2 (local) — 컴포넌트 안에 갇힘
  예: error variant 이름, internal helper 분해, 한 함수 시그니처
L3 (trivial) — 노트 불필요
  예: 변수명, import 순서, 주석
